# Brainstorm — 구현 전 기획 검증

구현을 시작하기 전에 요구사항을 철저히 분석하고, 사용자에게 핵심 질문을 던져 기획의 빈틈을 메운다.

## 인자

- `$ARGUMENTS`: 구현할 기능 또는 주제 (필수)

## 실행 절차

1. **주제 분석** — `$ARGUMENTS`를 분석하여 기능의 범위, 영향, 의존성을 파악한다.

2. **기존 코드 탐색** — 관련 파일, 패턴, 유사 구현을 찾아 컨텍스트를 수집한다.

3. **질문 생성** — 사용자에게 **최소 10개의 질문**을 던진다:
   - 기능 요구사항 명확화 (무엇을, 왜)
   - 기술적 제약 (호환성, 성능, 보안)
   - 엣지 케이스 (에러 상황, 경계값)
   - 의존성 (다른 기능/서비스와의 관계)
   - 우선순위 (MVP vs 전체 구현)
   - UX 기대 (사용자 관점)

4. **사용자 응답 수집** — 답변을 기반으로 추가 질문하거나 정리한다.

5. **기획서 작성** — 수집한 정보를 아래 형식으로 정리한다:

```markdown
## 기획서: {기능명}

### 목표
{한 문장으로 정리}

### 범위
- 포함: {구현할 것}
- 제외: {구현하지 않을 것}

### 기술 설계
- 아키텍처: {선택한 접근 방식과 근거}
- 영향 범위: {수정할 파일/모듈 목록}
- 의존성: {필요한 외부 라이브러리/서비스}

### 에지 케이스
- {식별된 에지 케이스 목록}

### 검증 계획
- {테스트 전략, 성공 기준}

### 작업 분해
1. {단계 1}
2. {단계 2}
...
```

6. **사용자 승인** — 기획서를 사용자에게 보여주고 승인을 받는다.

## 핵심 원칙

- "일반 자연어"보다 **개발/아키텍처 지식을 주입**하는 방향으로 질문한다
- 검증 Plan은 최소 **85% coverage**를 목표로 한다
- 사용자의 생각과 **"일치"**하는지 적극적으로 확인한다

## 사용 예시

```
/brainstorm 사용자 인증 시스템 구현
/brainstorm React Flow 캔버스에 undo/redo 기능 추가
/brainstorm Plane 웹훅 연동으로 실시간 이슈 동기화
```
